## 大白话串讲：Agent 怎么安全地做事 + 花钱

昨天解决了「Agent 能拿什么权限」，今天解决两个问题：**Agent 拿到权限后怎么一步步做事**，以及**怎么花钱不乱花**。最底层，Account Abstraction 提供了链上的技术底座。

---

### 一、Agent Workflow — 让概率模型走确定性流程

> **核心：不要把完整执行链路藏在一段长 prompt 里，要拆成每一步都可检查、可暂停、可恢复的固定流程。**

#### 为什么需要这个

昨天说了 Agent 是概率系统。概率系统的危险在于——它可能在某一步「发挥过度」：跳过风险检查直接发交易、把越权请求当成正常任务、在交易 pending 时又发一笔。

Workflow 做的就是：**用确定性的流程框住概率模型。** 模型负责在每个节点里做判断，但模型不能决定自己下一步去哪。

#### 一条完整的 Task Graph（以 swap 为例）

```
用户目标：「帮我评估并执行一次低风险 swap」
  │
  ├─① 读取用户目标和限制        ← 自动
  ├─② 获取余额和 allowance       ← 自动（只读链上数据）
  ├─③ 查询价格和流动性          ← 自动
  ├─④ 生成候选交易              ← 自动
  ├─⑤ 模拟交易                  ← 自动
  ├─⑥ 展示风险                  ← 展示给人看
  ├─⑦ 用户确认                  ← ⚠️ 人必须确认
  ├─⑧ 发送交易                  ← 自动执行
  └─⑨ 追踪结果                  ← 自动
```

**每一步都有输入、输出、可用工具和失败处理。** 第⑤步模拟失败了，不是继续往前走，而是退回到③重新查价格。第⑧步交易 pending 了，不能盲目再发一笔。

#### 三层风险，三层确认

Handbook 给了一个极清晰的分层，这就是昨晚你总结的那句话的展开版：

| 风险等级 | 操作类型 | 谁来确认 |
|----------|----------|----------|
| 🟢 低 | 只读分析 | 自动执行 |
| 🟡 中 | 小额白名单操作 | Session Key 自动执行 |
| 🔴 高 | 高风险交易 | **必须人工确认** |
| ⛔ 禁止 | 超出 policy | 直接拒绝，不经过人 |

**关键点：人确认的时候不能只看一个「确认」按钮，必须能看懂资产变化、权限变化和失败风险。** 否则人确认就变成形式主义了。

#### 状态机 — 系统不能忘记自己在哪

传统 chatbot Agent 的状态藏在聊天记录里。用户刷新页面、RPC 超时、交易 pending——系统就「失忆」了。

Workflow 要求显式状态：

```
draft → context_loaded → plan_ready → simulation_failed → waiting_user_confirmation
→ submitted → confirmed / reverted / cancelled
```

不管出什么问题，系统始终知道自己当前在哪一步、下一步该往哪走。

#### Trace — 出事了不能只看聊天记录

> 「没有 trace，出了问题只能看聊天记录；有 trace，才能复盘是模型理解错、工具返回错、policy 漏了，还是用户确认了错误动作。」

一条合格的 trace 包含：用户目标 → 模型版本 → 上下文来源 → 工具输入输出 → policy 判断 → simulation 结果 → 人工确认 → 交易哈希 → 最终状态。

#### Regression Set — 每次改模型前先跑一遍

就像软件工程的回归测试，但测的是**安全性**：

- 正常 swap ✓
- 错误链请求 → Agent 应该拒绝
- 无限 approve 请求 → Agent 应该拒绝
- 余额不足 → Agent 应该停止
- 用户拒绝签名 → Agent 应该停止
- 预言机价格过旧 → Agent 应该警告

### 二、Machine Payment — Agent 花钱的核心不是转账，是预算+报价+收据

> **支付不是转账本身，而是谁授权、价格多少、预算多少、失败怎么退、收据怎么验证。**

#### 为什么 Agent 需要支付能力

Agent 要买的东西：模型推理、数据 API、浏览器环境、链上交易 Gas、存储、另一个 Agent 的任务执行。全是付费服务。只能免费调工具的 Agent 永远停在 demo 阶段。

#### 三条铁律

1. **预算先于执行** — 没有预算边界，就没有安全自动支付。不能先花钱再算账。
2. **报价必须可比较** — Agent 要知道价格、币种、有效期、服务范围、退款条件。过期报价不能用。
3. **收据必须可验证** — 付完钱要能证明付给谁、为什么付、交付了什么。

#### 一个完整支付链路

```
用户授权预算（今天最多 3 USDC）
→ API 返回 Quote（一次调用 0.1 USDC，5 分钟有效）
→ Agent 检查预算和服务方身份
→ 系统检查 Policy（是否在白名单、是否超限）
→ 付款
→ API 返回结果 + Receipt
→ 系统记录 Quote、Payment Intent、tx hash、结果、剩余预算
```

#### 预算怎么设才对

一个常见错误：只设总预算，不设频率和服务方范围。结果是 Agent 可以把全部预算花在一个异常报价上，或者被恶意网页诱导重复购买同一个服务。

正确做法：**多层预算**：
- 全局预算（今天最多 X）
- 任务预算（这次任务最多 Y）
- 单次调用上限（一次最多 Z）
- 服务方上限（某个服务方最多 W）
- 紧急停止（一键关停所有自动支付）

#### 稳定币适合机器支付，但要注意

稳定币价格稳定、可编程，但 Agent 不能只看到「0.1」，必须知道是 0.1 USDC 还是 0.1 USDT 还是 0.1 原生 token——每一条链、每一个 token address、每一个 decimal 都不一样。

#### Payment Intent — 把「想付钱」和「已付钱」拆开

这是一个很重要的概念。Payment Intent 说的是「用户授权这类付款」，不是「这笔钱已经扣了」。它在真正结算之前创建，用来给后续 payment、escrow 和 receipt 提供上下文。包含：用户目标、服务方、最大金额、币种、链、过期时间、是否允许重试、是否需要人工确认。

### 三、Account Abstraction — 所有这一切的链上底座

> **普通钱包 = 一把私钥控制一切。智能账户 = 账户自己是一段程序，可以写规则。**

#### EOA vs 智能账户

| | EOA（你的 MetaMask） | 智能账户（Account Abstraction） |
|---|---|---|
| 怎么控制 | 唯一一把私钥 | 可编程规则 |
| 权限粒度 | 有私钥 = 能做一切 | Session Key 可以只允许特定合约、额度、时间 |
| Gas 谁付 | 必须你自己有 ETH | Paymaster 可以代付 |
| 丢了怎么办 | 私钥丢了就没了 | 社交恢复、多签恢复 |
| Agent 能用吗 | 不能（太危险） | 可以（画个笼子） |

#### Session Key — Agent 的临时权限卡

**这就是 Agent Wallet 的技术实现。**

Session Key 不是主私钥。它有边界：只在某段时间有效、只能调某个合约、只能用某些方法、只能花某个额度、只能在特定链上执行。

> 「你不希望 AI Agent 每次都打断用户签名，也不希望它拥有无限权限。Session Key 提供中间状态。」

#### AA 不是让 AI 更自由

Handbook 收尾一句很关键：

> 账户抽象不是让 AI 更自由，而是让 AI 的自由被规则包起来。

这跟昨天 Agent Wallet 的核心认知完全一致：控制权不能交给概率系统。

### 四、全景：Day 1 + Day 2 拼图

```
第一层：链上规则
  Smart Contract ─── 资产和规则在链上，公开可验证
  Account Abstraction ─── 钱包不再是私钥，是可编程账户

第二层：Agent 能力边界
  Agent Wallet ─── Agent 只能拿到被限制、可撤销的权限空间
  Session Key ─── 技术实现：限时间、限额度、限合约

第三层：执行流程
  Agent Workflow ─── 任务拆步骤，每步可暂停、可恢复、可审计
  Human-in-the-loop ─── 高风险必须人确认，低风险自动走

第四层：支付
  Machine Payment ─── 预算→报价→支付→收据，全程可验证
  Payment Intent ─── 授权和结算拆开，不让 Agent 临时决定花钱
```

**一句话总结 Day 2：**

> Agent 不是拿到权限就放飞——Workflow 管它怎么一步步走，Machine Payment 管它怎么花钱，Account Abstraction 在链上提供可编程的规则底座。三者合在一起，Agent 才从「会聊天的脚本」变成「可控的自动化系统」。
