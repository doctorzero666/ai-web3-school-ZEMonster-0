## Notes

### Agentic Commerce — 机器买东西的六个环节

> Agentic Commerce = 把购买意图、预算、验收标准、争议规则变成结构化协议。Agent 可以替用户花钱，但不能替用户承担无限责任。

| 环节 | 知识节点 | 解决的信任问题 |
|------|---------|--------------|
| 买前 | **Payment Intent** | Agent 不能把钱花超/花错地方 |
| 买前 | **Budget Control** | 一个循环错误不会烧光余额 |
| 买中 | **Agents Purchasing APIs** | Agent 怎么动态发现靠谱服务 |
| 买后 | **Proof of Task Completion** | 服务方怎么证明任务完成了 |
| 买后 | **On-chain Receipt** | 双方事后怎么对账 |
| 买后 | **Escrow Flow** | 不信你也不信我，钱先锁着 |

### 五问拆解

> 场景：用户让 Agent 去购买「链上数据分析 API」

| 问题 | 答案 |
|------|------|
| 谁发起？ | 用户（说"分析这个地址过去 30 天的交互"） |
| 谁执行？ | Agent 发现 API → 询价 → 调用 |
| 谁付款？ | Agent 代用户，从预算中扣（稳定币/测试网） |
| 谁验证？ | 系统检查返回格式 + 请求 ID；用户抽查 |
| 谁担风险？ | 用户（有限）— 预算上限锁定，超时不付，结果不对可争议 |

### 最小流程

```
用户 → Payment Intent（5 USDC，只买数据分析 API，24h 有效）
         │
         ▼
Agent 发现服务 → 比较 3 个 provider
         │
         ▼
服务方报价（价格 / 返回格式 / 响应时间 / 退款条件）
         │
         ▼
Agent 在预算内调用 → 拿到结果
         │
         ▼
验证：JSON 格式正确？有请求 ID？数据非空？
         │
    ✅ 放款 + 链上收据
    ❌ 退款 + 记录失败原因
```

## Issues / Questions

- 如果 API 返回了格式正确但内容错误的数据（比如分析结果造假），Proof of Task Completion 怎么验？
- Payment Intent 的「有效期」过期后，已发起但未完成的任务应该自动取消还是等待人工确认？
