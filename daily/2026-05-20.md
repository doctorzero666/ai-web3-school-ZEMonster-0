## 预习：两个概念一句话

**Settlement & Escrow**：不是「把钱打过去就完了」，而是把资金先锁在合约里，等服务交付并被验收后才释放——每一步有状态、有证据、有退款规则。

**Agent Identity**：不是给 Agent 起名字，而是让别人能验证「它是谁、谁控制它、能做什么、服务入口在哪、历史能不能追溯」。

## Notes

### Settlement & Escrow — 付完钱怎么确认交付

> 钱不是付了就完了，是先锁在中间人那里，等服务交付、验收通过，才放款。出问题了能退钱、能争议。

**比喻**：闲鱼交易。钱先打到支付宝，你收货、验货、点确认，钱才给卖家。不对就退款，扯皮就仲裁。

**六个状态**：
```
Created → Funded（钱锁进托管）→ Delivered（交付物提交）
→ Accepted（验收通过）→ Released（钱释放给服务方）
出问题：
→ Refunded（超时/失败退钱）→ Disputed（扯皮，仲裁介入）
```

**交付证明**：不是嘴上说做好了。得有 hash、时间戳。API 调用看请求/响应 hash，代码任务看 commit hash + 测试通过，数据任务看 dataset hash。

**验收**：低风险 AI 自动验（字段完整？格式对？），高风险必须人工复核。Handbook 建议三层：AI 初审 → challenge window → 人工复核。

**争议**：提前写好规则。谁能发起、押金多少、谁当裁判、裁决后能不能上诉。不能等出事了再讨论。

### Agent Identity — 付钱给谁

> 不是给 Agent 起名字，是让别人验证：它是谁、谁控制它、能干什么、入口在哪、历史能不能查。

**四个必须回答的问题**：
- Owner：谁控制它？（EOA / 多签 / DAO）
- Capability：它能做什么？（能力声明 + schema + 价格 + 限制）
- Service Endpoint：怎么调它？（API / MCP Server / A2A）
- 历史记录：之前靠谱吗？（交易记录、评价、争议）

**关键认知**：身份 ≠ 可信。身份只是给了你一个固定坐标系去追踪行为。靠谱不靠谱看 Reputation（后面要学的声誉系统）。

### 两节串起来：Agent 间委托任务完整链路

```
① Identity — A 在 Registry 发现 B，看 profile、能力、历史
② Payment  — A 拿到 B 的 Quote（价格、过期时间、交付条件）
③ Escrow   — A 把 2 USDC 锁进托管合约 → Funded
④ Work     — B 干活
⑤ Proof    — B 提交报告 + hash + 时间戳 → Delivered
⑥ Accept   — Evaluator 验收通过 → Accepted
⑦ Settle   — 托管释放 2 USDC → Released
⑧ Receipt  — 记录全程：谁付谁、金额、任务 ID、tx hash、结果 hash
```

失败了走 Refunded / Disputed 分支。
