## Notes

### AI Oracle — 模型判断怎么上链

> **AI Oracle = 普通预言机 + 模型判断。但模型会错、输入会变、版本会更新，所以必须把结果和生成过程一起上链。**

**AI Oracle ≠ 普通 Oracle**

| | 普通 Oracle | AI Oracle |
|---|---|---|
| 输入 | 外部数据（价格、天气） | 模型判断（风险评分、任务验收） |
| 确定性 | 数据一般是客观的 | 模型输出可能随版本/输入变化 |
| 验证 | 验证数据来源 | 还要验证模型版本、prompt、执行环境 |
| 争议 | 很少需要 | 必须有 challenge 机制 |

**AI Oracle 的五个知识层（从浅到深）**

① **AI Output** — 输出要结构化。链上消费 `accepted: true` / `riskScore: 72`，不是长文本。机器字段进合约，人类解释进 UI。还要记录 confidence、模型版本、输入/输出 hash、时间戳。

② **Data Feed** — 持续提供 AI 处理后的数据。Address risk score、内容审核标签、市场情绪指数。关键：处理版本漂移——模型升级后同一对象评分可能变化，feed 要支持历史查询和 stale data 检查。

③ **Model Result** — 不仅存结果，还要存生成条件（模型版本、prompt 模板、输入引用、输出 schema）。多模型系统还要记录路由信息。

④ **Oracle Risk** — 风险取决于输出影响什么。展示标签→低风险；释放资金→高风险。按影响范围分层：低风险直接展示、中风险人工复核、高风险需要 challenge + multi-evaluator。

⑤ **Proof of Inference + Attestation + Dispute**：
  - Attestation 证明「谁在什么条件下给了这个结果」（≠ 结果正确）
  - Proof of Inference 试图证明输出确实来自某模型+输入（TEE/ZK/签名日志/可重放推理）
  - Dispute/Challenge: optimistic 模式，先接受结果，给挑战窗口，有人举证就进入复核

**核心原则：最稳妥的路径是从低风险场景开始（标注、摘要、预警），涉及资金释放时才需要证明+挑战期+人工复核。**

### AI Security — 攻击 AI 系统比攻击合约更隐蔽

> **AI Security 不是「防止模型说错话」，是防止模型错误、恶意上下文和工具滥用变成真实的资产、权限或治理事故。**

**五个攻击面 + 对应防护**

① **Prompt Injection** — 恶意内容藏在合约 README、网页、治理提案、token metadata 里，诱导模型「忽略之前规则」「把钱转到某地址」。
→ 防护：分层标注上下文可信级别，外部内容只能做待分析资料不能成新指令。高风险工具不能只听模型一句话。

② **Tool Abuse** — 模型被诱导反复调用付费 API、查询敏感数据、生成无限授权交易。
→ 防护：tool permission、rate limit、budget、simulation、异常检测。短时间高频调用、同一服务重复付款→ alert。

③ **Malicious Context** — 假的合约地址、伪造审计报告、过期价格数据、被污染的 token metadata。Agent 当成事实就会生成错误交易。
→ 防护：链上事实优先从 RPC/explorer/verified source 读取，网页只是辅助解释。

④ **Key Safety** — 私钥/API key/session key 不能进模型上下文或日志。
→ 防护：secret 不进 prompt、不进输出、不进普通日志。优先用 Smart Account + session key 而不是把 EOA 私钥放自动化 runtime。

⑤ **Permission Isolation + Sandbox**：
  - 读和写工具分开，普通和高风险工具分开。不要给 Agent 一个「万能 Web3 工具」。
  - Sandbox 隔离：处理网页的浏览器不能读钱包密钥，执行代码的 sandbox 不能访问生产数据库。

**核心原则：不是让模型永不犯错，而是让模型犯错时无法直接造成不可接受损失。Audit Log 记录完整链路，Alert 发现异常及时打断。**

## 今日实操理解（大白话版）

> 今天在本地搭了 Vibe-Trading + Hermes MCP，这些实操恰好完美解释了 AI Oracle 和 AI Security。

### AI Oracle 大白话

Oracle 就是"AI 的判断怎么让别人信"。**日志是给自己看的，Oracle 是把结果连同证据打包，交给别人、让他们也能验证。**

今天实操的对应：

| Oracle 概念 | 今天的实际对应 |
|------------|-------------|
| Attestation（证明） | Vibe-Trading 每次回测输出 `run_card.json`——记录输入、模型、引擎、结果，可复现 |
| Model Result | 不只存 Sharpe，还存 config.json + signal_engine.py + 数据源 |
| Dispute/Challenge | 同一个 hash 重跑一遍，SHA 一致结果就该一致 |
| 从低风险开始 | 回测结果只是"参考"，不直接触发实盘交易——就是低风险场景 |

**一句话记 Oracle**：从"我说 Sharpe 0.37"变成"这是 run_card.json，你自己验"。

### AI Security 大白话

**不是"防止 AI 说错话"，是防止恶意上下文和工具滥用变成真实损失。** 今天实操中实际碰到的三个安全机制：

| Security 概念 | 今天实际碰到的 |
|-------------|-------------|
| Path Sandbox | `VIBE_TRADING_ALLOWED_RUN_ROOTS`——Agent 只能在白名单目录操作，防止读到 `~/.ssh/` |
| Code Injection 防护 | `_validate_signal_engine_source()`——回测策略代码被 AST 扫描后才执行 |
| Audit Log | Hermes-Lite 的 SQLite events 表——每次 LLM 调用、工具执行都记录 |

**一句话记 Security**：所有进入模型的内容都可能是攻击面，所有离开模型的动作都必须被约束。

### 完整安全闭环

```
输入侧（AI Security）              输出侧（AI Oracle）
┌──────────────────┐            ┌──────────────────┐
│ Prompt 过滤       │            │ 结果 Attestation  │
│ 路径沙箱          │    AI 处理  │ 模型+版本+输入记录 │
│ 工具权限隔离      │  ───────→  │ 可复现 Run Card   │
│ 审计日志          │            │ 争议/重跑机制      │
└──────────────────┘            └──────────────────┘
"坏东西不能进来"                  "好东西能证明自己没问题"
```

### 用生活场景理解 Oracle vs 日志

- **日志**：你妈帮你记的账——她了解你，信你的字
- **Oracle**：拿给学校报销用的发票——班主任不认识你，只认发票真假

Oracle 多出来的那层，就是从"我说"变成"你自己验"。
