## Notes

### Agent Trust & Reputation — 怎么知道 Agent 靠不靠谱

> 不是看自我介绍，是看历史行为、谁验证过、押了多少钱、失败过几次。

**关键认知**：

① **声誉要按任务类型拆开。** 擅长总结治理提案的 Agent 不一定擅长写合约测试。五星好评没用，要绑定具体任务 ID、交付物 hash、验收标准。

② **Stake 让承诺有成本。** Agent 自称厉害——那你押点钱，失败了罚没。但 stake ≠ 能力，有钱不一定强。所以要和 review、validation、历史记录一起看。

③ **防刷。** 自己开小号互刷是真实风险。评价者本身也需要身份；用户应能看到评价者和被评价者之间的交易关系。

④ **时间衰减。** 两年前好不代表现在好——模型可能换了、owner 可能换了、endpoint 可能换了。

**核心原则**：信任不是一个分数，是一组可追溯、可比较、可解释的证据。

**ERC-8004**：把 Agent 的 identity、reputation、validation 分成可组合 registry，不试图做成单一中心化评分。链上注册证明身份连续性，不能自动证明服务质量。

### Verifiable AI — 怎么证明 AI 做的事是真的

> AI 输出会影响资产和权限时，不能只靠「我相信模型」，得能验证输入、模型、执行环境和结果。

**三层验证强度**：

| 场景 | 验证方式 | 成本 |
|------|----------|------|
| 总结新闻 | 签名日志（Audit Trail） | 几乎免费 |
| 风险评分 | TEE（可信执行环境） | 中等 |
| 释放 escrow | ZK proof（密码学验证） | 贵但最强 |

**五种工具，从便宜到贵**：

① **Audit Trail** — 最基础、最容易落地。记录输入、输出、模型版本、时间、用户确认、交易哈希。不能证明模型绝对正确，但能证明系统当时看到了什么。出事可复盘。

② **TEE** — 硬件隔离环境运行程序，发 attestation 证明。适合「不能公开输入但需要证明在受保护环境里跑过」。缺点：仍依赖硬件和供应链信任，不是纯密码学。

③ **ZK / zkML** — 密码学验证计算满足条件，不泄露全部输入。最强但最贵。完整证明 LLM 推理还不现实，目前适合证明关键步骤或小模型。

④ **Attestation** — 某主体对 Agent 或结果的可验证声明。需要 issuer、证据、过期时间、撤销机制，否则会误导人。

⑤ **Benchmark + Regression** — 公开 benchmark 说明一般能力，但 Agent 需要自己的任务集：正常样本 + 攻击样本（错误链、恶意上下文、过期价格、假合约文档）。

### 信任体系完整链条

```
Agent Identity    ──→ 知道是谁
Reputation        ──→ 知道靠不靠谱（历史行为 + stake + review）
Verifiable AI     ──→ 用技术手段证明靠不靠谱不是吹的
```

三者缺一不可：没有 Identity 找不到对象，没有 Reputation 不敢付钱，没有 Verifiable AI 出事了没法追责。
