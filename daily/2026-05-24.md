## Notes

### AI Privacy — 模型看到的数据越少，泄漏面越小

> **AI Privacy ≠ 数据加密存。是在每一层决定：哪些数据进模型、哪些只进工具、哪些只在本地、哪些绝不外传。**

**AI Privacy vs AI Security 的分工**

```
AI Security（Day 5） → 防止攻击从外面进来（注入/滥用/恶意上下文）
AI Privacy（Day 6）  → 防止数据从内部出去（钱包历史/私钥/偏好泄漏）
```

**核心风险：链上公开 + AI 私有 = 去匿名化**

```
单看链上地址 0x123... → 只是一串字符（公开但无意义）
+ Agent 知道你的交易习惯 → "常用 Uniswap，偏好低滑点"
+ Agent 知道你的聊天内容 → "上周说想买某 NFT"
+ Agent 知道你的风险偏好 → "保守型，止损很紧"
────────────────────────────────────────
= 完整的真实身份画像 + 交易策略 + 心理弱点
```

**七个知识节点（从基础→高级）**

```
Data Boundary          ← 初：数据在哪、怎么流、谁看得到
  ├── Minimal Disclosure   ← 初：证明"够付"不需要暴露全部持仓
  ├── User Consent         ← 初：同意连钱包 ≠ 同意分析全部历史
  ├── Local AI             ← 中：敏感数据本地处理，摘要才发云端
  ├── Private Memory       ← 中：Agent 记的东西必须可查看可删除
  ├── Encrypted Data       ← 中：加密存 ≠ 模型用的时候不会泄漏
  └── Secret Management    ← 高：私钥/API key 绝不进 prompt
```

**Secret Management — 最强硬的一条**

> 助记词和私钥不应该出现在任何 Agent prompt。即使用户主动粘贴，系统也应该识别并拒绝处理。

为什么这条单独拎出来？因为 Web3 场景里私钥泄漏 = 所有资产归零，不像传统 App 的 API key 泄漏还能轮换。

**和前面几天的关系**

| 前面学的 | AI Privacy 给的约束 |
|---------|-------------------|
| Day 1: Chain-aware Context（读链上数据） | 读了但不能把所有地址都关联到同一身份 |
| Day 1: Agent Wallet（Agent 管钱） | 管钱时不能把钱包历史全塞进第三方模型 |
| Day 3: Agent Identity（Agent 是谁） | Identity 和真实身份不能默认绑定 |
| Day 3: Settlement & Escrow | 交易的对手方信息也是隐私 |

**大白话记 Privacy**

> 你家门牌号公开（链上地址），但你在家说什么、买什么是私密的。不要因为 Agent 进了你家，就把日记连同门牌号一起发到网上。原则：**最小数据、可删可查。**

### Governance AI — 帮人看懂提案，不替人投票

> **Governance AI 的核心，是提高公共决策的信息质量，而不是替代人的政治判断。**

**为什么需要这个**

DAO → 200个提案 + 50场会议 + 30笔预算申请 + 海量论坛讨论
    → 少数人掌握上下文 → 多数人只能被动跟投
    → AI 可以整理，但必须保留来源、区分事实和判断

**七个知识节点（从基础→高级）**

```
基础层（必须做到）:
  Proposal Summary       ← 把长提案结构化：目标/背景/预算/风险/待决
  Meeting Action         ← 会议 → 可执行事项（谁、做什么、deadline）
  Source Traceability    ← 每条结论必须附来源链接（治理 AI 的底线）

进阶层:
  Budget Check           ← 预算是否清晰、里程碑可验证、是否有重复拨款
  Contribution Graph     ← 发现"隐形贡献"（协调/文档/新人引导）

高级层:
  Deep Funding           ← 公共物品资助：影响非线性，不能用简单 KPI
  Plurality              ← 保留多元视角，不把 LP/开发者/持币者压成一个意见
```

**Proposal Summary — 不是把提案缩成宣传稿**

好的摘要结构：
```
✅ 提案目标 + 背景
✅ 预算金额 + 付款地址 + 里程碑
✅ 支持理由
✅ 反对理由        ← 这条最容易被 AI 省略，但最关键
✅ 未回答问题
✅ 相关历史提案
⚠️ "以上为 AI 摘要，不构成投票建议" ← 必须声明
```

**Source Traceability — 治理 AI 的绝对底线**

> 没有来源的治理摘要，不能用于投票依据。

每条关键结论都要能点回原文。如果 AI 给不出来源 → 标注"AI 推断"。

**和前面几天的关系**

| 前面学的 | Governance AI 怎么用 |
|---------|-------------------|
| Day 5: AI Oracle（验证结果上链） | 上链后的提案/预算，AI 帮你解读 |
| Day 4: Agent Trust & Reputation | Contribution Graph 喂给 Reputation |
| Day 4: Verifiable AI | Proposal Summary 也要可验证（来源反查） |
| Day 2: Machine Payment | Budget Check + 链上付款地址验证 |

**大白话记 Governance**

> 一个 DAO 有 200 个提案，你不可能全看完。AI = 帮你整理好关键信息、标注来源、列出正反理由。但最后那个"赞成/反对"按钮——你自己点。

                Oracle 输出的数据进入 DAO 提案/预算/投票流程

## 最小实践

### 1. Agent 数据流图

```
用户输入 ──→ [本地脱敏] ──→ 摘要 → 云端模型
                   │
钱包地址 ──→ [只读工具] ──→ 链上数据 ──→ Agent 上下文
      ✗ 不进第三方模型         │
                   │
API key ──→ [Secret Manager] ──→ 工具调用
      ✗ 不进 prompt / 不进日志 / 不进模型

每层标注：
  - 数据存在哪（本地 / 服务器 / 链上）
  - 谁看得到（用户 / Agent / 第三方模型 / 公开）
  - 怎么删除（用户操作 → 立即生效 / N天后自动清理 / 链上不可删）
```

### 2. 治理提案摘要模板

```
## [提案名称]
- 目标：
- 背景：
- 预算：XX USDC → 地址 0x...（多签？/ EOA？）
- 里程碑：
  1. [ ] M1: ... 截止: ...
  2. [ ] M2: ...
- 支持理由：
- 反对理由：
- 未回答问题：
- 相关历史：
- ⚠️ 以上为 AI 摘要，不构成投票建议
  每条结论见来源：[链接]
```
