## x402 协议 — 「把付款塞进 HTTP 请求里」

### 一句话

> x402 = HTTP 402 Payment Required 状态码的复兴。把链上支付嵌入 HTTP 协议层，让机器买东西像人刷卡一样自然。

### 大白话讲一遍

你（的 Agent）去调一个 API，对方返回的不是数据，而是一个 **402 Payment Required**，附带一行 `X-402-Payment: {"network":"base","token":"USDC","amount":"0.5"}`。

你看懂了：哦，要付 0.5 USDC。于是你签一笔链上交易，把付款凭证附在重试请求里。对方验证链上到账了，给你数据。

整个过程：**没有注册账号、没有 KYC、没有 API Key、没有预充值、不用填信用卡**。

### 为什么叫 x402

HTTP 协议从 90 年代就留了 402 这个坑——"Payment Required"，但从来没被真正用过。x402 把它填上了：响应头里塞链上支付信息，客户端付款后重试。

### 对比传统方式

| | 传统 SaaS API | x402 |
|---|---|---|
| 注册 | 填邮箱、密码、手机号 | 不需要 |
| 付费 | 填信用卡 / 预充值 | 链上转账 USDC |
| 安全 | 管理 API Key，怕泄露 | 不需要 Key，每次请求带付款证明 |
| Agent 能用吗 | 需要人帮忙注册绑卡 | Agent 自己完成全套 |
| 退款 | 联系客服 | 协议层面还没解决，看 ERC-8183 |

### 生产数据（2026年5月，过去30天）

- 7,541 万笔交易
- $2,424 万交易额
- 9.4 万买家，2.2 万卖家

已经在跑了，不是白皮书。

### 在六个节点中的位置

```
x402 覆盖：
  ✅ Agents Purchasing APIs  — 这是 x402 的主场
  ✅ Payment Intent         — 请求里带预算信息
  ❌ Budget Control         — x402 不管，应用层做
  ❌ Proof of Completion    — x402 不管，ERC-8183 管
  ❌ On-chain Receipt       — x402 不管
  ❌ Escrow Flow            — x402 不管，一次付清

x402 的本质：买 API 的「支付通道」，解决怎么付的问题，
          不解决「付了之后拿不到货怎么办」的问题。
```

## ERC-8183 协议 — 「Agent 任务的托管结算机」

### 一句话

> ERC-8183 = 一个极简的链上状态机。用户锁钱 → 服务方干活 → 评估人验收 → 放款或退款。

### 六个状态

```
        创建任务
           │
           ▼
    ┌──────────┐  没钱 / 取消   .─────────────.
    │   Open   │──────────────▶│   Rejected   │
    └────┬─────┘               `─────────────'
         │ 用户打钱
         ▼
    ┌──────────┐  评估人拒绝    .─────────────.
    │  Funded  │──────────────▶│   Rejected   │
    └────┬─────┘               `─────────────'
         │ 服务方提交成果
         ▼
    ┌──────────┐
    │Submitted │
    └────┬─────┘
         │
    ┌────┴────┐
    ▼         ▼
Evaluator  Evaluator
批准        拒绝
    │         │
    ▼         ▼
Completed  Rejected
放款给      退款给
服务方      用户
```

六个状态：**Open → Funded → Submitted → Completed / Rejected / Expired**

| 状态 | 含义 | 谁能操作 |
|------|------|---------|
| Open | 任务创建了，但还没打钱 | 用户可以取消 |
| Funded | 钱锁在合约里了 | 服务方可以提交成果；评估人可以拒绝 |
| Submitted | 服务方交活了 | **只有评估人**能批准或拒绝 |
| Completed | 放款给服务方 | 终态 |
| Rejected | 退款给用户 | 终态 |
| Expired | 超时退款给用户 | 终态（`expiredAt` 到了，任何人可触发） |

### 三个角色

| 角色 | 干什么 | 类比 |
|------|--------|------|
| **Client（用户）** | 创建任务、打钱 | 你 |
| **Provider（服务方）** | 干活、交活 | API 服务商 |
| **Evaluator（评估人）** | 验收，决定放款还是退款 | 裁判 |

**核心设计**：Evaluator 可以是用户自己、一个智能合约、或第三方。如果是智能合约，可以自动检查：
- 返回格式对不对
- 数据哈希对不对
- 零知识证明过没过

### 一个完整的例子

```
你让 Agent 买链上数据分析 API：

1. 创建 Job（ERC-8183）
   {
     client:    你的地址，
     provider:  数据分析服务方地址（可后期指定），
     evaluator: 你的 Agent 的合约地址（自动验收），
     budget:    5 USDC，
     expiredAt: 24 小时后
   }

2. 状态 = Open
   你调用 fund() → 5 USDC 从你钱包转入合约
   状态 = Funded

3. Agent 通过 x402 调用服务方 API
   服务方干活，拿到分析结果

4. 服务方调用 submit(deliverable)
   状态 = Submitted

5. 你的 Evaluator 合约自动检查：
   - JSON 格式正确？
   - 有请求 ID？
   - 数据哈希匹配？
   → 全部通过，调用 complete()
   → 状态 = Completed
   → 5 USDC 从合约转给服务方
```

### 和 x402 的关系

```
x402 = 买 API 时的「付钱通道」
     = 肉身支付层

ERC-8183 = 买之前锁钱 + 买之后验收 + 不满意退款
         = 托管结算层

两者互补：
  x402 管「怎么付」
  ERC-8183 管「付了之后怎么保证不亏」
```

## 把它们串回六个知识节点

| 节点 | x402 | ERC-8183 | 还缺什么 |
|------|:---:|:---:|------|
| Payment Intent | 🟡 请求里带 | 🟢 结构化存链上 | — |
| Budget Control | ❌ | 🟢 budget 字段 + expiredAt | 失败预算需应用层 |
| Agents Purchasing APIs | 🟢 主战场 | ❌ | 服务发现还需注册表 |
| Proof of Task Completion | ❌ | 🟢 Evaluator + attestation | — |
| On-chain Receipt | 🟡 交易哈希即收据 | 🟢 Job 本身就是收据 | — |
| Escrow Flow | ❌ | 🟢 完整状态机 | — |

🟢 = 协议直接解决 | 🟡 = 部分覆盖 | ❌ = 不在范围内

## 关键认知

1. **x402 不是支付方案，是支付标准。** 它不管你用 USDC 还是 ETH，不管你在 Base 还是 Arbitrum。它只管 HTTP 402 这个协议层约定。

2. **ERC-8183 的极简是故意的。** 只有 6 个状态、3 个角色、一个 ERC-20。复杂的逻辑留给 Evaluator 合约，不塞进协议本身。可组合 > 大而全。

3. **Evaluator 是可编程的信任。** 小交易 Evaluator = 用户自己（人工看），大交易 Evaluator = 智能合约（自动验证），最大交易 Evaluator = 第三方仲裁（人类陪审团）。
